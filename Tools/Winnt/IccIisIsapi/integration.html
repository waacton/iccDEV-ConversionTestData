IIS Integration Guide

This page turns the operator notes into a website-style runbook. It is the same guidance captured in isapi-instructions.md, repackaged for browser use inside the deployed IIS sample.

Quick start

  1. Build and install the IIS sample into out\iis-shared-install\bin.
  2. Point an IIS site at that `bin` directory.
  3. Allow iccIisIsapi.dll in ISAPI restrictions and map it as a handler.
  4. Open `./` for the landing page, then confirm the DLL endpoints return `200`.

Expected routes

`/`
Landing page, served by index.html.
`/iccIisIsapi.dll`
Version summary.
`?mode=health`
Plain-text health response.
`?format=xml`
Minimal ICC XML.
`POST ?mode=tools&input=icc|xml`
Runs the four native tools and returns JSON for the browser console.

Shared-library deployment

The site folder must contain the handler DLL plus the runtime DLL set that IIS needs to load it successfully.

  • iccIisIsapi.dll
  • IccProfLib2.dll
  • IccXML2.dll
  • iconv-2.dll
  • libxml2.dll
  • zlib1.dll
  • iccDumpProfile.exe, iccToXml.exe, iccFromXml.exe, iccRoundTrip.exe for the browser tool console
  • This website shell: index.html, endpoints.html, integration.html, site.css, site.js, assets\*

The installed web.config also maps .icc and .icm to application/octet-stream so uploaded profiles and regenerated profiles can be downloaded directly from the per-request workspace URLs.

Browser tool suite

The endpoint console now supports raw file upload from the browser. Upload an ICC profile or ICC XML file on Endpoint Console and IIS will run iccToXml, iccFromXml, iccDumpProfile, and iccRoundTrip inside the worker process environment.

The response is JSON and the page renders the real exit code, console output, generated XML preview, and any tool-specific warnings. Each request also leaves behind a per-request workspace under _tool-work\<job> so the uploaded input, generated XML or ICC, and tool logs can be fetched directly over HTTP. Round-trip failures are surfaced per tool rather than being treated as an HTTP failure.

Why HTTP 403.14 happens

`403.14 - Forbidden` at the site root means IIS is serving a directory path while directory browsing is disabled and no default document is configured for that path.

The fix for this sample is to keep directory browsing off, leave web.config in the site root, and serve index.html as the default document. That is why this local site now opens cleanly at `/`.

Operator checklist

  • IIS `ISAPI Extensions` feature is enabled.
  • The app pool architecture matches the build architecture.
  • The site physical path points at the installed `bin` directory.
  • The ISAPI restriction entry explicitly allows iccIisIsapi.dll.
  • The site handler list includes a mapping for iccIisIsapi.dll using `IsapiModule` and allows POST as well as GET/HEAD.
  • The runtime DLLs are present next to the handler.
  • The four IIS-exposed tools are present next to the handler.

curl.exe http://localhost:18081/iccIisIsapi.dll?mode=health

Representative build and install flow

if not defined VCPKG_ROOT echo Set VCPKG_ROOT first and exit /b 1

set "PREFIX=%CD%\out\iis-shared-install"
set "TOOLCHAIN=%VCPKG_ROOT%\scripts\buildsystems\vcpkg.cmake"

cmake -S Build\Cmake -B out\iis-shared-vcpkg-toolchain ^
  -G "Visual Studio 17 2022" ^
  -A x64 ^
  -DCMAKE_TOOLCHAIN_FILE="%TOOLCHAIN%" ^
  -DCMAKE_INSTALL_PREFIX="%PREFIX%" ^
  -DVCPKG_TARGET_TRIPLET=x64-windows ^
  -DENABLE_TESTS=OFF ^
  -DENABLE_TOOLS=ON ^
  -DENABLE_WXWIDGETS=OFF ^
  -DENABLE_SHARED_LIBS=ON ^
  -DENABLE_STATIC_LIBS=ON ^
  -DCMAKE_INTERPROCEDURAL_OPTIMIZATION=OFF ^
  -Wno-dev

cmake --build out\iis-shared-vcpkg-toolchain --config Release ^
  --target iccIisIsapi iccIisIsapiSmoke -- /m /maxcpucount
cmake --install out\iis-shared-vcpkg-toolchain --config Release

powershell -ExecutionPolicy Bypass -File Tools\Winnt\IccIisIsapi\Install-IccIisIsapiSite.ps1 ^
  -SiteName "Codex-iccIisIsapiInstall" ^
  -Port 18081

FAQ

  • Do I need directory browsing? No. The correct fix is a default document.
  • Can I test without IIS? Yes. Use iccIisIsapiSmoke.exe for a local DLL load test.
  • Where do I check live behavior? Use the Endpoint Console for both GET checks and file uploads.
